# Click VPN — Site-to-Site и продвинутая маршрутизация
## Дизайн-документ (design doc)

> Статус: **черновик / проектирование**. Код не начат.
> Цель документа — зафиксировать архитектуру до реализации, чтобы не
> переписывать всё на каждом этапе.

---

## 1. Цели и не-цели

### Цели
- Связывать **сети целиком** (площадки/филиалы), а не только отдельных клиентов.
- Поддержать **несколько транспортов** туннелей: WireGuard, OpenVPN, IPsec, GRE.
- **Статическая** и (позже) **динамическая** маршрутизация (FRR: BGP/OSPF).
- **Гибкая матрица доступа** — кто из площадок видит чьи подсети (сегментация).
- Реализовать как **управляемый модуль** (вкл/выкл), не раздувая ядро.
- Генерировать готовые конфиги для роутеров площадок (MikroTik и пр.).

### Не-цели (сознательно вне scope первых этапов)
- Полный mesh «все со всеми напрямую» (для серых IP нереалистичен — звезда).
- Автоматическое применение конфига на чужие роутеры (панель только генерирует).
- L2-растяжение сетей между площадками (только L3-маршрутизация).
- Графический drag-and-drop редактор топологии (позже, если будет нужно).

## 2. Топология

**Hub-and-spoke (звезда).** Хаб = VPN-сервер с панелью (белый IP). Спицы =
роутеры филиалов (часто за серым IP/NAT). Трафик между филиалами идёт через хаб.

```
        ┌─── Площадка А (LAN 192.168.10.0/24)
        │
   ХАБ (VPN-сервер) ─── Площадка Б (LAN 192.168.0.0/24)
        │
        └─── Площадка В (LAN 192.168.200.0/24)
```

Почему звезда: спицы за NAT не могут принимать входящие; хаб с белым IP —
единственная точка, к которой все подключаются. Хаб выполняет маршрутизацию
и применяет матрицу доступа.

---

## 3. Архитектура — три слоя

### Слой 1. Транспорт (туннель) — абстракция протокола
Единый интерфейс, конкретные реализации-плагины. Ядро site-to-site не знает
деталей протокола.

```
class Transport (ABC):
    name: str                       # 'wireguard' | 'openvpn' | 'ipsec' | 'gre'
    def hub_apply(hub, sites): ...   # поднять/обновить туннели на хабе
    def hub_teardown(hub): ...
    def site_config(hub, site): ...  # сгенерировать конфиг для роутера площадки
    def status(): ...                # состояние туннелей (up/down, handshake)
```

Реализации:
- `WireGuardTransport` — на базе services/wireguard.py (есть)
- `OpenVPNTransport` — ccd + iroute (есть база OpenVPN)
- `IPsecTransport` — strongSwan swanctl, conn type=tunnel (есть база IKEv2)
- `GRETransport` — Linux `ip tunnel` (gre/gretap), обычно GRE over IPsec

Добавление нового протокола = новый класс. Регистрируется в реестре транспортов.

### Слой 2. Маршрутизация
Не зависит от транспорта. Два режима:

**Статическая (этапы S1–S2):**
- Хаб знает LAN каждой площадки → форвардит между ними (ip_forward + FORWARD).
- Каждой площадке выдаются маршруты к LAN других (через её туннель к хабу).
- В терминах протоколов: WG `AllowedIPs`, OpenVPN `iroute`+`push route`,
  IPsec `local_ts/remote_ts`.

**Динамическая (этап S5):**
- FRRouting на хабе и (опц.) на площадках. iBGP или OSPF поверх туннелей.
- Маршрутизаторы обмениваются маршрутами автоматически — не прописывать руками
  при добавлении подсети. GRE-туннели несут протоколы маршрутизации.

### Слой 3. Матрица доступа (ACL)
Таблица `site × site → allow/deny` (направленная или симметричная).
Реализация двумя уровнями:
1. **Видимость маршрута** — площадка получает в `AllowedIPs`/маршруты только
   те подсети, к которым ей разрешён доступ.
2. **Принуждение на хабе** — iptables FORWARD-правила: даже если маршрут есть,
   хаб дропает запрещённый трафик (defense-in-depth).

Пример: А↔Б разрешено, В видит только сервер головного, А↔В запрещено.

## 4. Модель данных

```
Site (площадка)
  id, name, role (hub|spoke), transport (wireguard|openvpn|ipsec|gre),
  endpoint (для hub — публичный host:port; spoke — пусто/серый),
  wg_private_key, wg_public_key,           # для WG-транспорта
  psk / certs ...,                          # для IPsec
  tunnel_ip (адрес в туннельной сети, напр. 10.100.0.X),
  created_at

SiteSubnet (LAN-подсеть площадки)
  id, site_id, cidr (192.168.10.0/24), comment
  # уникальность CIDR между площадками — проверять (конфликт ломает маршрутизацию)

AccessRule (матрица доступа)
  id, src_site_id, dst_site_id, allow (bool)
  # отсутствие правила = политика по умолчанию (allow_all | deny_all из настроек модуля)

# Конфиг модуля (Module.config JSON):
  hub_site_id, tunnel_network (10.100.0.0/24), default_policy, routing_mode (static|dynamic)
```

## 5. Поток работы (user flow)

1. Включить модуль «Site-to-Site».
2. Создать **хаб** (обычно = текущий VPN-сервер, белый IP, endpoint).
3. Добавить **площадки** (спицы): имя, LAN-подсети, транспорт.
4. Настроить **матрицу доступа** (или оставить «все видят всех»).
5. Панель:
   - на хабе поднимает туннели/пиры, настраивает форвардинг + iptables по матрице;
   - для каждой площадки генерирует **конфиг роутера** (скачать/QR/ссылка).
6. Админ заливает конфиг на роутер филиала (MikroTik и т.д.).
7. Туннель поднимается, подсети видят друг друга по матрице.
8. Статус туннелей виден в панели (up/down, handshake, трафик).

## 6. UI

Раздел «Площадки» (появляется при включённом модуле):
- **Карта/список площадок** — статус туннеля, LAN-подсети, транспорт.
- **Матрица доступа** — таблица site×site с чекбоксами allow/deny.
- **Добавить площадку** — мастер (имя, транспорт, подсети).
- **Конфиг площадки** — скачать/показать (для роутера филиала) + инструкция.
- Индикатор конфликтов подсетей.


Все — под guard `modules.is_enabled('site2site')`.

## 8. Поэтапный план

| Этап | Содержание | Результат |
|---|---|---|
| **S0** | Модуль site2site, модели Site/SiteSubnet/AccessRule, абстракция Transport, реестр | каркас |
| **S1** | WireGuardTransport (звезда), статическая маршрутизация, генерация конфигов площадок, форвардинг на хабе, UI площадок + статус | **рабочий s2s на WG** |
| **S2** | Матрица доступа (AllowedIPs по правам + iptables на хабе), проверка конфликтов подсетей | сегментация |
| **S3** | OpenVPNTransport + IPsecTransport (плагины) | мульти-протокол |
| **S4** | GRETransport (over IPsec) | носитель для динамики |
| **S5** | Динамическая маршрутизация (FRR: BGP/OSPF), автообмен маршрутами | продвинутое |

Каждый этап самодостаточен и тестируется отдельно.

## 9. Модуляризация существующих функций

Параллельно навести порядок: крупные опциональные функции — в управляемые модули
(как биллинг). Ядро (серверы, клиенты, статистика, безопасность) — всегда.

Кандидаты в модули:
- `billing` — уже модуль ✅
- `site2site` — новый
- `installers` — Windows .exe + изолированная раздача ссылок (share-сервис)
- `expiry_notify` — уведомления об истечении сертификатов

Реестр модулей расширяется записями; включение/выключение управляет видимостью
разделов и фоновых процессов. on_disable снимает эффекты (как у биллинга).

## 10. Риски и нюансы

- **Конфликт подсетей**: две площадки с одинаковым CIDR ломают маршрутизацию —
  обязательная проверка уникальности при добавлении.
- **Туннельная адресация**: выделить служебную сеть (напр. 10.100.0.0/24) под
  адреса туннелей площадок; не пересекать с LAN филиалов.
- **Серые IP спиц**: только исходящие; PersistentKeepalive (WG) / DPD (IPsec)
  держат туннель за NAT. Failover между WAN — на стороне роутера (watchdog).
- **Применение на роутеры**: панель генерирует конфиг, заливает человек.
  Авто-провижн чужих роутеров — вне scope.
- **Производительность хаба**: весь межфилиальный трафик идёт через хаб —
  он не должен быть бутылочным горлышком (сетевой канал/CPU).
- **MTU/фрагментация**: туннели уменьшают MTU; GRE+IPsec особенно — учитывать
  mss-clamping, чтобы не было чёрных дыр PMTUD.
- **iptables vs nftables**: на новых Debian — nftables; команды должны работать
  через iptables-nft (обычно прозрачно).
- **Безопасность матрицы**: enforcement обязательно на хабе (iptables), не только
  через видимость маршрутов — иначе обход правкой конфига на стороне площадки.

## 11. Принятые решения (зафиксировано)

1. **Несколько хабов** (multi-hub) — гибче. В модели `Site.role=hub` + у спицы
   `hub_id`. Хабы могут связываться между собой (hub-to-hub) для крупных сетей.
   *Поэтапно:* S1 реализует один хаб (MVP), multi-hub — после single-hub.
   Модель и абстракция сразу закладывают multi-hub, чтобы не переписывать.
2. **Туннельная сеть настраиваемая** — задаётся per-hub в конфиге модуля
   (по умолчанию подсказка 10.100.0.0/24); валидация на непересечение с LAN.
3. **Политика матрицы настраиваемая** — `default_policy ∈ {allow_all, deny_all}`
   в конфиге модуля; AccessRule переопределяет дефолт точечно.
4. **Динамическая маршрутизация — BGP** (FRR). Обоснование: route-maps/политики
   ложатся на матрицу доступа, лучше масштаб, перспективнее для overlay.
   OSPF оставляем опцией для простых плоских топологий (флаг routing_proto).
5. **Только L3** (маршрутизация подсетей). L2/GRETAP — вне scope (риск штормов,
   MTU); задел в абстракции транспорта оставлен, но не реализуется в S0–S5.

### Влияние на модель/конфиг
```
Site: + hub_id (для spoke → к какому хабу; для hub → null), role, ...
Module.config (site2site):
  default_policy: 'allow_all' | 'deny_all'
  routing_mode:  'static' | 'dynamic'
  routing_proto: 'bgp' | 'ospf'        # при dynamic
  # туннельная сеть — per-hub (в записи Site хаба), не глобально
Site(hub): + tunnel_network (напр. 10.100.0.0/24), + bgp_asn (для dynamic)
```
