# Click VPN — Селективная маршрутизация (split tunnel / обход блокировок)
## Дизайн-документ

> Статус: **проектирование**. Код не начат.
> Цель: гнать в туннель только нужные ресурсы (заблокированные сайты/сервисы),
> остальной трафик — напрямую. Дополняет уже готовую обфускацию (AmneziaWG / TLS-crypt).

---

## 1. Цель и ключевая проблема

Маршрутизация работает по **IP**, а сайты-сервисы — по **доменам**, причём крупные
живут на **CDN** с сотнями меняющихся диапазонов. Наивный `resolve(domain) → /32`
даёт неполное покрытие: часть контента приходит с других edge-узлов CDN, уходит
мимо туннеля → сайт «частично не работает» или блокировка не обходится.

**Вывод:** список маршрутов нужно собирать из НЕСКОЛЬКИХ надёжных источников
(в первую очередь — опубликованные/анонсируемые диапазоны CDN), агрегировать в
минимальный набор CIDR и периодически обновлять.

---

## 2. Источники маршрутов (RouteSource)

Профиль маршрутов наполняется записями нескольких типов; компилятор сводит их в
итоговый набор CIDR.

| Тип | Что это | Покрытие CDN | Стабильность |
|---|---|---|---|
| `provider` | встроенные фетчеры опубликованных диапазонов: Cloudflare (`/ips`), Google (`goog.json`), AWS (`ip-ranges.json`), Telegram (CIDR-лист), Meta, Apple | полное | высокая |
| `asn` | все анонсируемые префиксы автономной системы сервиса (AS15169 Google, AS32934 Meta, AS62041 Telegram, AS13335 Cloudflare…) через RIPEstat/bgp.tools | полное | высокая |
| `url_list` | внешний готовый список CIDR по URL (antifilter.download, re:filter — community-списки заблокированных ресурсов РФ) | широкое | обновляется автором |
| `domain` | резолв A/AAAA домена → /32,/128; опц. расширение до /24; накопление по нескольким резолвам | частичное | низкая |
| `cidr` | ручной статический CIDR/подсеть | точечно | ручная |

**Принцип:** для крупных сервисов опираемся на `provider`/`asn` (полное и стабильное
покрытие CDN), `domain` — только как дополнение для мелких сайтов, `url_list` — для
готовых списков обхода.

## 3. Конвейер компиляции

```
источники профиля
   → fetch/resolve каждого (с кэшем и таймаутами)
   → объединение всех IP/CIDR (v4 и v6 раздельно)
   → агрегация: ipaddress.collapse_addresses (слить смежные/вложенные)
   → отсев мусора (0.0.0.0/0, приватные, дубли)
   → compiled_cidrs (кэш в БД) + метка времени и счётчик префиксов
```

- **Агрегация обязательна** — без неё тысячи /32 раздувают AllowedIPs/таблицу
  маршрутов. `collapse_addresses` даёт минимальный набор префиксов.
- **Лимит префиксов** (напр. ≤ 5–10 тыс.) с предупреждением в UI: на телефоне
  слишком большой AllowedIPs замедляет старт WG.
- **Фоновое обновление** (services/routelists.py): раз в 12–24 ч перекомпиляция;
  кнопка «обновить сейчас». provider/asn-источники меняются редко → конфиги
  клиентов не нужно дёргать часто.

---

## 4. Модель данных

```
RouteProfile (профиль маршрутов)
  id, name, mode (selective | full | exclude),
  dns_through_tunnel (bool), compiled_cidrs (TEXT, кэш), compiled_at, prefix_count

RouteSource (источник в профиле)
  id, profile_id, kind (provider|asn|url_list|domain|cidr),
  value (имя провайдера | ASN | URL | домен | CIDR), enabled

# Назначение профиля:
VPNUser.route_profile_id  (или серверный дефолт VPNServer.route_profile_id)
```

`mode`:
- `selective` — в туннель идут ТОЛЬКО CIDR профиля (AllowedIPs = набор).
- `full` — всё через туннель (0.0.0.0/0) — как сейчас.
- `exclude` — всё через туннель, КРОМЕ CIDR профиля (split исключением; для
  «локальный трафик мимо VPN»). Реализуется вычитанием из 0.0.0.0/0.

## 5. Применение в конфиг (генераторы уже есть)

**WireGuard / AmneziaWG** (`services/wireguard.build_client_conf`):
- `selective`: `AllowedIPs = <compiled_cidrs>`
- `exclude`: `AllowedIPs = 0.0.0.0/0` минус CIDR (предвычесть в компиляторе)
- `dns_through_tunnel`: добавить IP DNS в AllowedIPs + строка `DNS =`.

**OpenVPN** (`profile_builder`):
- `selective`: набор `route <net> <mask>` в клиентском конфиге, БЕЗ
  `redirect-gateway`. DNS — опц. `dhcp-option DNS`.
- сервер: маршруты/NAT уже есть (MASQUERADE pool).

**Роутеры (MikroTik/Keenetic/OpenWRT)** — опционально, для доменной точности:
генерировать скрипт address-list + routing-mark, который роутер обновляет по
расписанию (там доменный split реально точен через ipset+DNS).

---

## 6. DNS — отдельный важный момент

При блокировках часто травится и DNS. Поэтому:
- опция **DNS через туннель**: IP резолвера добавляется в AllowedIPs, в конфиг
  пишется `DNS = <resolver>`. Иначе резолв пойдёт мимо и сайт не откроется даже
  при верных маршрутах.
- свой резолвер на хабе (dnsmasq/unbound) — опц. этап, для доменной точности.

---

## 7. Корректность / типичные грабли

- **Неполное покрытие** → блокируемый IP уедет напрямую. Лечится provider/asn
  источниками (полные диапазоны) + регулярным обновлением.
- **Смена IP сервисами** → для стоковых WG-клиентов AllowedIPs статичен; опираемся
  на стабильные provider/asn-диапазоны, перевыпуск конфига при больших изменениях.
- **IPv6**: если у клиента есть IPv6, а в наборе только v4 — трафик уйдёт по v6
  мимо. Либо включать v6-диапазоны, либо глушить v6 у клиента.
- **Размер AllowedIPs** на мобильных — держать агрегированным и под лимитом.
- **DNS-leak** — см. п.6.

## 8. UI

Модуль `selective_routing` (вкл/выкл, как billing):
- Страница «Маршруты»: список профилей; внутри профиля — источники (добавить
  provider/ASN/URL/домен/CIDR), режим, DNS-опция, статус компиляции
  (когда обновлён, сколько префиксов), кнопка «Пересобрать».
- Пресеты-категории (под капотом — набор источников): «Telegram», «YouTube/Google»,
  «Соцсети (Meta)», «Discord», «Обход РФ (re:filter)», «свой список».
- В карточке клиента — выбор профиля маршрутов (по умолчанию `full`).

## 9. Поэтапный план

| Этап | Содержание | Результат |
|---|---|---|
| **R0** | Модуль + модели RouteProfile/RouteSource + компилятор (cidr/domain/url_list) + агрегация | каркас, ручные списки работают |
| **R1** | Применение в WG/AmneziaWG (AllowedIPs selective/exclude) + UI профилей + назначение клиенту | рабочий split-tunnel на WG |
| **R2** | provider-фетчеры (Cloudflare/Google/Telegram/Meta/AWS) + asn-источник (RIPEstat) + фоновое обновление | надёжное покрытие CDN |
| **R3** | OpenVPN (push route без redirect-gateway) + DNS через туннель + пресеты-категории | мульти-протокол + DNS |
| **R4** | Скрипты для роутеров (MikroTik/Keenetic address-list) для доменной точности | доменный split на роутерах |
| **R5** | Свой резолвер на хабе (unbound) + анти-DPI ротация порта/домена | продвинутое |

## 10. Принятые решения (зафиксировать)

1. Основной путь — **списки IP/CIDR** (работают со стоковыми клиентами), НЕ
   привязка к клиенту Amnezia. Доменная точность — опционально через роутеры (R4).
2. Покрытие CDN — через **provider + asn** источники (полные диапазоны), domain —
   вспомогательно.
3. Компилятор обязательно **агрегирует** (collapse_addresses) и кэширует результат.
4. Режимы: selective / full / exclude. По умолчанию у клиента — full (как сейчас).
5. DNS-through-tunnel — опция профиля (иначе DNS-leak/непрохождение).
6. Модуль управляемый (`selective_routing`), как billing/site2site.
